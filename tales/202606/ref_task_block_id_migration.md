# Plan: Migrate `[[ref]]` Note `^task` Block IDs to `^ref`

## Context

Every `[[ref]]` note generated by `bob highlights` carries a generated PDF reading-status task line ending in the stable
block ID `^task`, for example:

```md
- [ ] #task [[lib/docs/obsidian_bases.pdf]] [p::2] ^task
```

The request is to rename that block ID to `^ref` so the status affordance of a `[[ref]]` note is named after the note
type itself. This mirrors the in-flight `bob projects` epic (`sdd/epics/202606/bob_projects_command.md`), which gives
`[[project]]` notes a `^prj` status task — after this migration each typed note's status task uses a type-specific block
ID (`^ref` for refs, `^prj` for projects) instead of the generic `^task`.

This is not a CLI surface change: no subcommands or flags are added, so the CLI-rules long-term memory does not need to
be consulted for this work.

### Current state — code

- `src/native/highlights_ref/mod.rs` centralizes the convention in `const PDF_TASK_BLOCK_ID: &str = "^task"` (line 56).
  The constant drives:
  - `parse_pdf_task_line()` (~line 3398) — finds the generated line via `contains_markdown_token`, which matches `^task`
    only as an exact whitespace-delimited token (so similarly named IDs can never be confused);
  - the `existing_annotation_task_identity()` guard (~line 3198) that excludes the generated line from annotation-task
    identity scans;
  - `default_note_body()` (~line 4473) — emits the block ID when creating a new reference note.
- Four error-message strings hardcode the literal `^task` (lines ~3410, ~3440, ~4592, ~4627).
- Test literals: unit tests in `src/native/highlights_ref/mod.rs` (~lines 6090–6901) and integration tests in
  `tests/cli.rs` (15 occurrences, e.g. lines 1607, 2686, 2953, 2998–2999, 3118, 3200, 3381, 3481, 3647, 3705, 3893,
  4212, 4464).
- Living docs: `README.md` (lines 176, 180) and `docs/highlights-ref-sync.md` (lines ~270, 385, 433, 450, 464, 668).
- Historical SDD records (`sdd/tales/`, `sdd/prompts/`, `sdd/research/`, prior `sase_plan_*.md`) also mention `^task`;
  those are immutable history and must NOT be rewritten.

### Current state — vault (surveyed today)

- 23 notes define a `^task` block, all of them `type: "[[ref]]"` notes (16 `chat`, 3 `papers`, 2 `docs`, 1 `blogs`, plus
  `ref/docs/obsidian_docs.md`), all under `~/bob/ref/`. No non-ref note defines `^task`.
- 33 `#^task` references live in exactly 9 daily notes (`~/bob/2026/20260603.md` through `20260611.md`), in three
  syntaxes: link `[[ref/...#^task]]`, embed `![[ref/...#^task]]`, and aliased link `[[ref/...#^task|title]]`. Every
  reference targets one of the 23 ref notes.
- No note anywhere in the vault uses a `^ref` block ID, so there is no collision.
- No non-markdown vault file (canvas, `.obsidian` config, templates) references `^task`.
- The vault contains unrelated block IDs `^tasks-in-pdfs`, `^task-toggle`, and `^task-keymap` that must be left
  untouched; all migration patterns must be anchored so they cannot match these.

## Scope

1. Change `bob-cli` so generated reference notes use `^ref` and parsing recognizes `^ref` as the generated PDF task line
   block ID.
2. Update tests, README, and `docs/highlights-ref-sync.md` accordingly.
3. Migrate the live vault: rewrite the 23 `^task` definitions to `^ref` and the 33 daily-note references to `#^ref`.
4. Verify both sides, then commit the touched vault files via the SASE commit workflow.

Out of scope: rewriting historical SDD records that mention `^task`; the `^prj` project-note work (separate epic);
keeping a permanent `^task` parse alias (see cutover rationale below).

## Design

### Code: hard cutover, single constant

Change `PDF_TASK_BLOCK_ID` from `"^task"` to `"^ref"`. Because parsing, the annotation-task guard, and note generation
all flow through the constant, behavior is otherwise unchanged. Update the four hardcoded error messages to say `^ref`
and show `^ref` in their example lines.

No transitional dual-acceptance of `^task` is needed. A version-skew window (old binary + migrated vault, or new
binary + unmigrated vault) is benign, not corrupting: `parse_pdf_task_line()` simply reports the generated line as
missing, which means no status signal is contributed and annotation-task creation fails loudly; existing note bodies are
never regenerated or auto-inserted (only the managed highlights region is rewritten), so nothing is duplicated or lost.
The code change and vault migration land together in this one unit of work, so the skew window only lasts until the
rebuilt `bob` binary is installed.

### Vault migration: anchored, token-safe rewrites

- Definitions (23 ref notes): on lines that are generated PDF task lines (checkbox + `#task` + PDF wikilink), replace
  the trailing whitespace-delimited token `^task` at end of line with `^ref` (pattern ` \^task$` → ` ^ref`). Any line
  ending in `^task` that does not match the generated-line shape is reported for manual review instead of silently
  rewritten.
- References (9 daily notes): replace only the two anchored forms `#^task]]` → `#^ref]]` and `#^task|` → `#^ref|`. This
  covers links, embeds, and aliased links, and cannot match the unrelated `^task*` block IDs.
- The rewrite is idempotent: a second pass finds nothing to change.

## Implementation Steps

1. In `src/native/highlights_ref/mod.rs`, change `PDF_TASK_BLOCK_ID` to `"^ref"` and update the four error-message
   literals (multiple-generated-lines error, malformed-line error, and the two missing-line errors).
2. Update unit-test literals in `src/native/highlights_ref/mod.rs` and integration-test literals in `tests/cli.rs` from
   `^task` to `^ref`.
3. Update `README.md` and `docs/highlights-ref-sync.md` so every generated-line example and contract description shows
   `^ref`.
4. Run focused tests (`cargo test highlights_ref`, plus the touched `tests/cli.rs` cases), then the full suite.
5. Check `git -C ~/bob status --short` to note pre-existing dirty files, then run the vault migration:
   - rewrite the 23 definition lines (` \^task$` → ` ^ref`) in the surveyed `ref/` notes;
   - rewrite the `#^task]]` / `#^task|` references in the 9 daily notes.
6. Verify the vault:
   - zero remaining `^task` end-of-line tokens and zero `#^task` references anywhere in the vault;
   - exactly 23 notes now define `^ref` (same file list as the pre-migration survey) and the daily-note reference count
     is unchanged at 33, now pointing at `#^ref`;
   - `^tasks-in-pdfs`, `^task-toggle`, and `^task-keymap` are untouched;
   - no files outside the 23 ref notes and 9 daily notes were modified by the migration.
7. Optionally sanity-check with `bob dataview` that the ref-note task lines still surface as tasks.
8. Commit only the touched vault files using the SASE git commit workflow, leaving unrelated pre-existing vault changes
   unstaged.
9. After the code lands, reinstall the deployed `bob` binary (e.g. `cargo install --path .`) so any cron or keymap
   invocations of `bob highlights` run with `^ref` expectations matching the migrated vault.

## Risks and Mitigations

- **Version-skew window**: a `bob highlights` run between vault migration and binary reinstall sees no generated task
  line — it contributes no status signal and errors only if it has new annotation tasks to insert. This is loud,
  idempotent, and self-heals after reinstall. Mitigation: do steps 5–9 back-to-back.
- **Over-broad replacement**: naive `^task` → `^ref` substitution would corrupt `^tasks-in-pdfs`, `^task-toggle`, and
  `^task-keymap`. Mitigation: only the anchored patterns above; verify those three IDs still exist post-migration.
- **User-authored `^task` lines that don't match the generated shape**: reported for manual review, never silently
  rewritten.
- **Obsidian Sync churn**: the live vault may pick up unrelated changes mid-run. Mitigation: snapshot
  `git -C ~/bob status` before and after, stage only the 23 + 9 touched files for the vault commit.
- **Broken Obsidian block links**: definitions and references are migrated in the same pass, so every `[[...#^ref]]`
  link resolves immediately after the rewrite; the collision check confirmed `^ref` was previously unused.

## Acceptance Criteria

- Newly generated reference notes contain `- [ ] #task [[...pdf]] [p::2] ^ref` and `bob highlights` parses, toggles, and
  reports status via the `^ref` line exactly as it previously did via `^task`.
- Error messages, README, and `docs/highlights-ref-sync.md` consistently describe `^ref`.
- The full Rust test suite passes.
- The vault contains no `^task` block definitions or `#^task` references; all 23 ref notes and 33 references use `^ref`;
  the three unrelated `^task*` block IDs are unchanged.
- Touched vault files are committed via the SASE workflow; historical SDD records remain unmodified.
