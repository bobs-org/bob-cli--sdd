# Remove Highlights Ref Summary/My Notes Scaffold

## Context

`bob highlights-ref scan` writes new `~/bob/ref` notes through the same note rendering path used by
`bob highlights-ref sync`: `ParsedNote::render_body()` calls `default_note_body()` in `src/native/highlights_ref/mod.rs`
when the target note does not already exist. That default body currently emits:

- the note title
- a PDF wikilink
- `## Summary`
- `## My Notes`
- `## Highlights`
- the managed `<!-- highlights:begin -->` / `<!-- highlights:end -->` region

The requested product change is to stop auto-generating the empty `## Summary` and `## My Notes` sections in newly
created reference notes. The managed Highlights section should remain because it is the command-owned body region used
for sidecar highlight rendering and future idempotent updates.

Existing reference notes are handled differently: if a target note already exists, the command requires the managed
Highlights markers and replaces only that managed region. Manual content outside those markers is preserved. This change
therefore should not remove `## Summary` or `## My Notes` from existing notes that already contain those sections; it
only changes the scaffold for newly generated notes.

## Goals

1. Make new notes generated by `bob highlights-ref scan` omit the empty `## Summary` and `## My Notes` sections.
2. Keep `## Highlights` and the managed begin/end markers unchanged.
3. Preserve existing-note update behavior: manual sections outside the managed markers remain untouched.
4. Update tests and docs so the executable/documented contract matches the new generated note shape.

## Non-Goals

- Do not add a CLI option or compatibility flag.
- Do not migrate or rewrite existing `~/bob/ref` files outside the command's normal sync/update behavior.
- Do not change frontmatter projection, marker parsing, sidecar parsing, block IDs, or dirty-vault preflight behavior.

## Implementation Plan

1. Update the new-note body renderer.
   - Modify `default_note_body()` in `src/native/highlights_ref/mod.rs` to stop pushing `## Summary` and `## My Notes`.
   - Keep the body order as title, PDF wikilink, blank line, `## Highlights`, managed begin marker, rendered highlights,
     managed end marker.
   - Keep newline spacing stable so existing snapshot-style substring checks remain easy to reason about.

2. Adjust integration tests that assert or rely on the old scaffold.
   - Update the sidecar creation test in `tests/cli.rs` to assert the generated note contains `## Highlights` without
     the removed headings.
   - Add or strengthen assertions that newly generated notes do not contain `## Summary` or `## My Notes`, especially in
     a scan-backed test because the user specifically called out `bob highlights-ref scan`.
   - Update tests that currently insert manual content by replacing `## My Notes\n\n`; insert manual content outside the
     managed Highlights region using a stable remaining anchor such as `## Highlights\n\n`.
   - Keep assertions proving manual content outside the managed region is preserved and dirty-note detection still
     refuses to modify dirty vault files.

3. Update documentation.
   - Change `docs/highlights-ref-sync.md` where it says new generated notes include `## Summary`, `## My Notes`, and
     `## Highlights`.
   - Keep README unchanged unless inspection reveals it has the same explicit scaffold statement; current README text
     describes the managed region but does not enumerate the removed headings.

4. Verify with focused tests first, then a broader check if needed.
   - Run the affected highlights-ref test subset, likely `cargo test highlights_ref`.
   - If failures indicate broader CLI expectations, run `cargo test --test cli highlights_ref`.
   - Optionally run `cargo test` if the targeted run passes and the repo size/runtime is acceptable.

## Risks and Mitigations

- Risk: tests that modeled manual user sections by editing `## My Notes` could stop exercising manual-content
  preservation. Mitigation: move those edits to a stable non-managed location before `## Highlights` and keep explicit
  preservation assertions.
- Risk: removing headings from new notes could be confused with migrating existing notes. Mitigation: only change
  `default_note_body()`; leave existing-note `replace_managed_region()` behavior untouched.
- Risk: formatting regressions around the managed markers could affect idempotence. Mitigation: preserve the existing
  marker layout and verify repeat scan/sync reports `writes: none` or `notes_unchanged` as existing tests expect.
